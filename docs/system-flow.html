Polaris Point — Site Builder Flow

From lead pull to live chatbot conversation. Every box maps to one or more functions in the codebase.

build · deploy · live loop

1

You — in admin/index.html The internal builder. Pre-fills, configures, deploys.

Step 1

Pull a lead → click "Build Site"

resetSiteBuilder() wipes any stale state from a previous lead, then the intake auto-fills from _leadsCache: business name, phone, address, listing URL.

Step 2

Search & Auto-Fill Google Places

One click pulls live data from Google: name · phone · hours · photos · reviews (now with real per-review star ratings) · place_id (saved for the weekly review refresh cron).

Step 2.5 — fallbacks

No online footprint? Use these.

Three optional buttons that fill empty fields so you always have a presentable demo:

Use Template Defaults

Loads the 5 baseline services from the template's config.js + generic hero/about copy.

Compose with AI

POSTs to /api/generate (type site_compose), Haiku drafts services + hero + about + sample reviews.

Use Generic Photos

Per-industry Pexels picker (5-8 curated photos). Pre-assigns hero/about roles.

Steps 3-5 — intake form

Fill or refine the structured config

  • Step 3 · Content — hero, about, services, reviews (with stars), FAQs, team, blog
  • Step 4 · Notifications & Reporting ALL TIERS — lead email, CC, alert cadence, report recipient, owner notes
  • Step 5 · Chatbot & Brand Voice ORBIT & APEX — persona, voice, custom greeting, never-say list, always-suggest list, brand differentiators
Step 6

Generate config.js + set admin password

Generator emits flat fields plus three nested blocks: chatbot:, brand:, ops:. Set the Site Admin Password (or click Generate for a memorable random one like Polaris-7K3M-9X2).

click "Deploy Live Site"
2

Deploy One click fans out to two endpoints, then converges on a success card.

POST /api/deploy

Infra provisioning

  • Creates a new GitHub repo (pp-<slug>)
  • Copies template files + config
  • Creates Vercel project, triggers first build
  • Wires custom domain (if provided)
POST /api/sites

DB registration

  • Inserts row in sites table
  • Stores SHA-256 admin_password_hash
  • Saves slug, template, vercel_url, repo_url, domain
  • Marks status active, owner = signed-in user
Success card

Live site + handoff materials

Shown once after deploy completes:

  • Visit Site ↗ — public URL
  • Site Admin ↗ (gold, prominent) — where the client signs in
  • Plaintext password — copy now, hash stored in DB
  • GitHub Repo ↗

Also appears in the Sites tab as compact Site ↗ / Admin ↗ buttons per row.

hand off URL + password
3

Client — at <their-site>/admin Refines what you pre-filled. Same source of truth, different entry point.

Auth

Sign in

Their per-site password (or your MASTER_ADMIN_HASH as a universal override). SHA-256 hash verified against sites.admin_password_hash via /api/config?action=login.

13-section editor

shared/site-admin.js

The client can refine any field — content, photos, theme, AND the new sections you set during build:

  • 1-11 — Business info, photos, hero, stats, about, services, reviews, FAQs, areas, theme, team
  • 12 · Notifications & Reporting — lead email, alert cadence, report recipient
  • 13 · Chatbot & Brand Voice — persona, never-say, always-suggest, brand voice

Save → PUT /api/config?slug=… updates sites.site_config in the DB. Live within ~60 seconds (Vercel edge cache TTL).

visitor lands on live site
4

Visitor — on the live site What the customer sees. Driven entirely by the saved config.

Page render

Static template + config injection

shared/config-engine.js reads config.js (or fetches from /api/config on DB-backed sites) and injects every field via data-cfg / data-cfg-list / data-cfg-item-stars directives.

Reviews now render real per-review star ratings — the new data-cfg-item-stars="rating" directive draws N filled ★ + (5−N) empty ☆ from the rating field, and updates aria-label for screen readers.

Source of truth
